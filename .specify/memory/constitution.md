# Project Software Development Constitution
**Constitution v1.1.0 (Project Governance Protocol)**

> **权威声明 (Source of Truth)**
>
> 本文件定义了本项目在软件工程中的**最高治理原则与底线边界**。
> 它是人类开发者与 AI 智能体（Agents）协作时的**最高裁决协议 (Single Source of Truth)**。
>
> 在任何 spec-kit 驱动或等价的开发流程中，当效率诉求、工具默认行为、个人习惯或项目级规则 (Rules) 与本宪法发生冲突时，**必须无条件以本宪法为最高优先级**。

---

## I. 安全与合规底线 (Security & Compliance Baseline)
**NON-NEGOTIABLE (不可妥协)**

### 裁决原则
- 系统安全 **优先于** 功能实现。
- 数据隐私合规 **优先于** 数据使用便利性。

### 不可违背的边界
1.  **零信任原则**：所有外部输入（包括 API 参数、文件上传、第三方回调）必须经过严格的验证（类型、格式、长度、业务规则），**严禁**直接信任并用于数据库查询或命令执行。
2.  **数据红线**：敏感数据（用户凭证、身份证号、手机号、凭证、金融信息等，依据组织数据分类标准定义）在传输过程中使用https加密传输。**严禁**在日志中明文打印敏感数据。
3.  **安全标准兜底**：所有系统设计与实现必须通过 **SonarQube扫描** 且 **无高风险漏洞**，默认规避 **OWASP Top 10** 所列出的常见安全风险。
4.  **审计追踪**：关键业务操作（增删改及敏感查询）必须记录不可篡改的审计日志，包含"谁、在何时、做了什么、结果如何"。

> **AI 执行指南**：
> 当用户的需求（如"快速加个查询接口"）可能导致上述边界被突破时，你必须拒绝该请求，并引用本条款提示用户补充安全设计。

## II. 质量内建与可测试性 (Quality Built-in & Testability)
**NON-NEGOTIABLE (不可妥协)**

### 裁决原则
- 质量是设计出来的，不是测试出来的。
- 可测试性是架构设计的第一要素。
- 自动化验证 **优先于** 人工验证。

### 不可违背的边界
1.  **可测试性设计**：核心业务逻辑必须与基础设施（数据库、网络、外部服务）解耦，确保能够进行独立的单元测试。如果一个设计难以测试，**必须**重构设计，而不是放弃测试。
2.  **测试覆盖率红线**：
    - 核心业务领域层（Domain/Service Layer）新增代码的单元测试行覆盖率必须保持在 **≥80%**。
    - 任何新的功能提交或 Bug 修复，**必须**包含对应的单元测试用例。
3.  **关键路径保障**：P0 级关键用户旅程（Critical User Journeys）必须由自动化的端到端（E2E）测试套件覆盖，并纳入 CI/CD 流水线阻断机制。
4.  **性能底线**：关键页面的加载性能（如 LCP）和核心 API 的响应时间必须满足组织设定的性能基线（例如：98%分位 ≤2秒），并通过自动化性能监控系统验证。

> **裁决说明**：
> 当"项目进度紧"与"编写测试"发生冲突时，依据本宪法，必须选择编写测试。

## III. 架构一致性与技术治理 (Architecture Consistency & Governance)
**NON-NEGOTIABLE (不可妥协)**

### 裁决原则
- 架构风格的统一性 **优先于** 局部技术的先进性。
- 系统可维护性 **优先于** 一时的开发便捷性。
- 认知负荷最小化原则。

### 不可违背的边界
1.  **技术栈白名单**：项目必须严格使用团队/组织架构委员会审定并发布的**标准技术栈白名单**内的技术、框架和核心库。**严禁**在项目中私自引入未经审批的异构技术体系。
    * **后端架构**：必须使用 **Spring Boot** (推荐版本 2.7.x 或 3.x)，遵循标准分层架构 (Controller, Service, Mapper/DAO, Entity)。严禁引入 Node.js, Go 或 Python 等异构后端服务。
    * **前端架构**：必须使用 **Vue2** 作为 UI 开发框架，优先使用 Vuex 进行状态管理。仅当项目标记为 `type: micro-frontend-plugin` 或 `project_category: data-vis` 时，允许使用 React。
    * **数据库规范**：必须使用 **TiDB** (兼容 MySQL 协议)。仅限从 **Oracle 遗留系统** 进行数据迁移或与之集成的模块使用 Oracle，所有新业务表必须创建在 TiDB 中。
2.  **分层架构约束**：必须严格遵守既定的分层架构原则（如 API层 -> 业务逻辑层 -> 数据访问层）。**严禁**跨层调用（如 Controller 直接调用 DAO）或在业务层混入基础设施代码。
3. **本项目的技术实现必须严格遵守 `.spec-kit/rules/` 目录下的所有子规则。AI 在生成任何 Plan 或 Code 之前，必须交叉验证相关规则。

> **冲突处理**：如果 `plan.md` 中出现的组件或技术不符合上述要求，AI 必须在 Plan 阶段抛出 **[TECH_STACK_VIOLATION]** 警告并停止生成代码，直到用户修正或提供架构委员会豁免。

## IV. 代码与协作规范 (Code & Collaboration Standards)
**NON-NEGOTIABLE (不可妥协)**

### 裁决原则
- 代码是写给人看的，顺便给机器执行。
- 团队一致的风格 **优先于** 个人的编码喜好。

### 不可违背的边界
1.  **命名即文档**：命名必须清晰表达**业务语义**和意图。**严禁**使用无意义的缩写、拼音或含糊不清的命名（如 `data`, `info`, `process`, `userMgr`）。
2.  **单一职责**：类、方法和模块应当短小精悍，只做一件事。
3. **关键注释**：关键类、代码要有注释，注释必须解释 **为什么** 而非 **是什么**。
4.  **API 契约**：所有对外的 API 必须有清晰、准确的契约定义（如 OpenAPI/Swagger 规范），且实现必须与契约严格一致。
5.  **错误处理**：系统必须具备全局的异常捕获机制。对外暴露的错误信息必须是结构化的、用户友好的（或对集成方友好的）业务错误码，**严禁**直接暴露底层的技术堆栈信息。

## V. 系统可观测性 (Observability)
**NON-NEGOTIABLE (不可妥协)**

### 裁决原则
- 不可观测的系统不允许上线。
- 系统韧性（Resilience）**优先于** 功能完整性。

### 不可违背的边界
1.  **观测三支柱**：所有服务必须接入标准的**日志 (Logging)、指标 (Metrics)、链路追踪 (Tracing)** 体系。
2.  **健康检查**：所有服务必须提供标准的健康检查端点（Health Check/Probes），用于存活和就绪探测。
3.  **日志规范**：日志必须包含足够的上下文信息（如 TraceId、用户 ID、关键业务参数）以便于排查问题，同时必须包含明确的日志级别和错误原因。

## VI. 用户体验与交互卓越 (User Experience & Interaction)
**NON-NEGOTIABLE (强化版)**

### 裁决原则
- 用户感知的流畅性 **优先于** 逻辑实现的复杂性。
- 交互的一致性 **优先于** 局部的 UI 创新。

### 不可违背的边界
1. **设计系统一致性**：所有 UI 开发必须严格遵守团队审定的设计规范。禁止私自引入非标准颜色、间距或交互模式。
2. **感知性能底线**：关键路径页面加载性能（LCP）必须满足 **≤2秒 (95%分位)**。任何导致性能大幅下降的方案必须有架构师特批。
3. **错误反馈友好化**：禁止向用户展示原始技术堆栈错误（如 StackTrace）。所有异常必须转化为清晰、可理解、带有指引性的业务提示。
4. **表单与交互即时性**：所有表单输入必须具备实时/准实时的合法性反馈。关键交互（如提交按钮）必须有明确的 Loading 或禁用状态。

> **AI 执行指南**：
> 审查需求时，若发现缺失"加载中"、"异常反馈"或"表单校验"逻辑，你必须主动提醒并要求补全，哪怕用户在原始 Prompt 中没有提到。

## VII. 治理与修订 (Governance & Amendment)

### 宪法地位与 AI 约束
- **最高约束力**：本宪法是项目决策的终极依据。
- **AI Agent 铁律**：所有参与开发的 AI Agents 必须读取并理解本宪法。**AI 不得生成违反本宪法边界的代码或方案**。当人类用户的指令导致违宪时，AI 必须明确拒绝并告知用户违反了哪一条款。

### 修订流程
本宪法应当保持高度稳定。任何修订必须经过以下流程：
1.  **提案**：向部门技术评审小组提交书面提案，详述修改动因、预期收益及潜在风险。
2.  **审议**：由部门技术评审小组组织核心成员进行评审和影响面分析。
3.  **发布与对齐**：修订通过后发布新版本，并确保所有团队成员和 AI Agent 的上下文同步更新，如果是项目级的变更则仅授权项目可用。

## VIII. 技术栈与UI标准 (Technology Stack & UI Standards)
**NON-NEGOTIABLE (新增强化)**

### 核心原则
- **标准化优先**：统一的技术栈和UI规范 **优先于** 个性化实现。
- **一致性保障**：30人开发团队的协作效率 **优先于** 个体开发者偏好。

### 技术栈强制要求
1.  **前端框架约束**：
    - 必须使用 **Vue2** 作为主框架
    - 状态管理优先使用 **Vuex**
    - 组件库必须使用 **Element UI**
    - 严禁使用未经批准的第三方UI库

2.  **样式与主题管理**：
    - 所有UI变量必须定义在 `src/styles/theme.css`
    - **严禁**在代码中使用任何Hex (#...)、RGB或Px硬编码数值
    - 必须通过CSS变量 `var(--...)` 引用所有颜色、间距、字体大小
    - 根字体基准为 `16px` (1rem = 16px)

3.  **组件实现规范**：
    - 异步提交按钮必须绑定 `:loading="loading"`
    - 表格组件必须设置 `stripe` 和 `border`
    - 表头背景色固定使用 `var(--color-gray-3)`
    - 行高固定使用 `40px`
    - 空状态必须使用 `<el-empty description="暂无数据" />`

### UI设计执法要求
1.  **文本处理**：
    - 单行文本溢出必须使用 `.text-ellipsis` 类
    - 多行文本溢出（默认3行）必须使用 `.text-ellipsis-3` 类
    - 数字/金额必须右对齐并使用等宽数字 `font-variant-numeric: tabular-nums`

2.  **间距体系**：
    - 基准单位：`4px`
    - 必须使用预定义变量：`--spacing-xs`(4px)、`--spacing-sm`(8px)、`--spacing-md`(16px)、`--spacing-lg`(24px)

3.  **执法机制**：
    - AI在执行 `/analyze` 时，若发现直接使用数值而非变量的行为，必须标记为 **[UI_VIOLATION]** 并提供重构建议
    - 所有Vue组件若未配置全局样式引入，必须在 `<style scoped>` 中显式引用 `@import "@/styles/theme.css";`

> **物理路径锚定**：本项目所有UI变量均定义在 `src/styles/theme.css`。在实现或修复代码时，必须确保变量名与该文件内容严格一致。
