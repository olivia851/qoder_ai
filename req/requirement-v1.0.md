系统“好不好”评价指标体系需求文档

 

1. 需求背景

 

为全面评估部门内部系统的“好不好用”，我们从“安全、稳定、实用、好用、智能”五个维度出发，建立一套可量化、可追踪的指标体系。通过设计前端页面直观展示各系统在各维度的表现，并结合Min-Max归一化方法进行得分计算，帮助团队识别强弱项，持续优化系统质量。

 

2. 总体目标

 

1）建立覆盖“安全、稳定、实用、好用、智能”五个维度的指标体系

2）实现指标数据的自动化采集与人工录入相结合

3）基于Min-Max归一化方法计算各维度得分

4）设计并生成前端页面，展示系统在各维度的表现，并与部门平均得分对比

 

3. 指标体系概览

 

| ***\*维度\**** | ***\*指标类别\****                           |
| -------------- | -------------------------------------------- |
| 安全           | 系统在信创、等保、漏洞、安全事件等方面的表现 |
| 稳定           | 系统可用性、事故数、合规性等                 |
| 实用           | 系统访问量、业务负载、获奖情况等             |
| 好用           | 代码质量、响应时间、易用性、需求交付效率等   |
| 智能           | 智能体应用个数、覆盖度、使用率、用户接受度等 |

 

4. 指标定义与计算逻辑

 

4.1 指标结构

每个指标包含以下字段：

\- 指标名称

\- 计算公式

\- 单位

\- 指标性质（正向/负向）

\- 数据来源

\- 提取方式（自动/人工）

 

4.2 示例指标

 智能程度（智能体应用）

\- 计算公式：系统接入智能体的个数

\- 单位：个数

\- 指标性质：正向指标

\- 数据来源：人工统计，部内评审

 

 代码重复率

\- 计算公式：流水线跑出的重复率

\- 单位：百分比

\- 指标性质：负向指标

\- 数据来源：系统提取，王汀

 

注：完整指标列表请参考附件《指标》sheet，涵盖31项指标的定义、来源、性质等。

 

5. 数据采集方式

 

5.1 系统提取

1）脚本收集分析：智能使用率、接受度、系统访问次数等

2）系统自动提取：流水线、漏洞线上管理系统、运营平台、ITSM系统等

 

5.2 人工统计分析

1）指标负责人通过加工、汇总等方式，如待优化功能点数、按时解决率、好用标准覆盖率等

2）人工统计，如信创情况、获奖数量、等保级别、好用标准覆盖率等

 

6. 得分计算方法

 

6.1 单个指标

采用“Min-Max归一化”方法计算每个指标的相对得分：

\- 正向指标：(实际值 - 最小值) / (最大值 - 最小值)  100%

\- 负向指标：(最大值 - 实际值) / (最大值 - 最小值)  100%

例：

|       | 智能程度（个） | 得分 | 代码重复率 | 得分 |
| ----- | -------------- | ---- | ---------- | ---- |
| 系统A | 4              | 100  | 30%        | 75   |
| 系统B | 1              | 0    | 20%        | 100  |
| 系统C | 2              | 33   | 60%        | 0    |

系统A智能程度的得分=（4-1）/（4-1）*100%=100

 

6.2 大类得分

通过灵活设置权重，得出大类得分、总得分：

 

1）初始阶段：等权重法

大类得分 = Σ(指标得分) / 指标个数

总得分 = Σ(大类得分) / 5

 

2）组合赋权法：

先确定大类：安全：20%（基础保障）稳定：20%（基础保障）  实用：25%（业务价值）好用：20%（用户体验）智能：15%（创新加分）

 

再确定指标：

以“好用”为例：

| ***\*小类\**** | ***\*指标\****   | ***\*建议权重\**** | ***\*依据\****             |
| -------------- | ---------------- | ------------------ | -------------------------- |
| 代码质量       | 单元测试覆盖度   | 8%                 | 基础保障，但用户不直接感知 |
| 代码质量       | 代码重复率       | 7%                 | 同上                       |
| 系统质量       | 生产缺陷数       | 15%                | 直接影响用户体验           |
| 及时性         | 平均响应时间     | 10%                | 体验关键指标               |
| 及时性         | 最长响应时间     | 8%                 | 体验关键指标               |
| 及时性         | 按时解决率       | 12%                | 用户核心诉求               |
| 易用性         | 平均事件数（率） | 10%                | 反映易用性问题             |
| 易用性         | 待优化功能点数   | 8%                 | 用户反馈                   |
| 易用性         | 好用标准覆盖率   | 6%                 | 过程指标                   |
| 易用性         | 好用标准达成率   | 8%                 | 结果指标                   |
| 需求交付效率   | 需求交付平均时间 | 8%                 | 响应速度                   |

 

最终实操中指标权重可在部门内讨论后确定，并阶段性调整，要求指标可配置，历史可查阅。

 

7. 数据输出与可视化

 

7.1 页面展示

\- 五个维度：安全、稳定、实用、好用、智能

\- 每个维度展示：维度下对应的多个指标

 \- 系统当前得分

 \- 部门平均得分（用于对比）

 

7.2 数据表

\- 支持按系统、维度查看原始数据与得分（含历史各期次的）

\- 支持导出为Excel或图表

 

8. 技术实现建议

 

 8.1 前端

\- 使用 ECharts或其它工具实现前端页面

\- 提供数据录入表单（人工指标）

\- 支持多系统、多时点切换查看

 

 8.2 后端

\- 提供指标数据录入接口（人工）

\- 提供自动采集数据接口（定时任务）

\- 提供得分计算服务（定时触发或实时计算）

 

 8.3 数据库设计建议

\- 指标定义表（维度、指标名、公式、性质、来源等）

\- 指标数据表（系统、指标、时间、实际值）

\- 得分结果表（系统、维度、得分、时间）
